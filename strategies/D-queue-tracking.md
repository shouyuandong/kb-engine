# 策略 D：队列追踪型

## 适用场景

定期扫描外部新增 → 排队尝试 → 对比标称与实测 → 提炼原理 → 借鉴自研。持续追踪外部生态，不封存。

**典型场景**：开源 skill/工具追踪、竞品分析、技术方案选型、GitHub Trending 追踪

**核心特征**：队列流转驱动。核心价值是"标称 vs 实测"的对比——这是别人库里没有的东西。收集让 LLM 做，尝试必须人做。

## 六维度定义

| 维度 | 定义 |
|------|------|
| 生命周期 | 持续型，不封存（生态持续涌现） |
| 内容流入 | 定期扫描（GitHub Trending、社区推荐） |
| 核心产出 | "标称 vs 实测"对比 + 原理沉淀 + 带溯源的自研成果 |
| 谁写什么 | LLM 写收集箱草稿、更新追踪摘要；人写尝试记录、最终判断 |
| 时效处理 | 旧记录不覆盖，用三层结构（冻结尝试 + 追加更新 + 演进判断） |
| 退出机制 | 不退出。作者弃坑标"稳定但停更"，不推荐标"不推荐"，都不删 |

## 目录配置（逻辑名）

```json
{
  "directories": {
    "收集箱": { "maintainer": "llm_full", "writable": true },
    "待尝试": { "maintainer": "human", "writable": false },
    "尝试记录": { "maintainer": "human", "writable": false },
    "沉淀": { "maintainer": "llm_draft_human_review", "writable": true, "draft_dir": "沉淀/.草稿" },
    "我的成果": { "maintainer": "human", "writable": false }
  },
  "compile_rules": [
    { "input": "外部扫描", "output": "收集箱", "rule": "LLM 从 README 提取标称能力，生成草稿" },
    { "input": "尝试记录", "output": "沉淀", "rule": "人写尝试记录后，提炼最终判断和原理" }
  ]
}
```

## 状态流转（单向）

```
收集箱 → 待尝试 → 尝试记录 → 沉淀
```

不反向移动。尝试过觉得没用的，直接标 `不推荐` 进沉淀，不退回待尝试。每个文件在哪个目录，就代表它在哪个阶段。

## 编译规则

- **外部扫描 → 收集箱**：LLM 从 README/文档提取标称能力，生成草稿。LLM 全权写入收集箱。
- **尝试记录 → 沉淀**：人写完尝试记录后，LLM 协助提炼最终判断和原理拆解，进草稿区等人审核。

> 收集箱可以 LLM 写，尝试记录必须人写——真实感受 LLM 替代不了。

## 更新追踪机制（三层结构）

旧记录不覆盖。每个沉淀的项目用三层：

1. **尝试记录**（历史快照，冻结不改）
2. **更新追踪**（追加，不覆盖）：LLM 定期用 gh CLI 检测更新，追加到此处
3. **判断**（随更新演进）：每条带日期，最新判断覆盖旧判断的"当前有效性"，旧判断保留作为演化轨迹

## 重试判断标准

| 更新类型 | 要不要重试 | 理由 |
|---------|-----------|------|
| 修了你当时吐槽的痛点 | 值得 | 负面判断可能已失效 |
| 新增功能 | 只测新功能 | 不重跑全部 |
| 纯重构/内部优化 | 不用 | 对使用者无感 |
| 大版本号变更 | 值得 | 可能有 breaking change |
| 作者弃坑/长期无更新 | 标"稳定但停更" | 不投入注意力但不等于没价值 |

## Lint 检查项

- 收集箱积压超过设定周期（如 30 天）未处理的
- 待尝试队列中超过设定周期没写尝试记录的
- 沉淀的条目是否都有最终判断
- 更新追踪是否长期未执行
- 尝试记录是否一直没写（收集箱→待尝试后卡住）
